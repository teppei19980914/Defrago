# MindFlow 要件定義書 - 非機能要件

更新日: 2026-03-03

---

## 1. 概要

本書は MindFlow アプリケーションの非機能要件を定義する。
品質特性（ISO 25010 ベース）に基づき、MECE に分類する。

### 1.1 要件分類体系

```
NFR: 非機能要件
├── NFR-PER: 性能
├── NFR-REL: 信頼性
├── NFR-USA: 使用性
├── NFR-SEC: セキュリティ
├── NFR-MNT: 保守性
├── NFR-PRT: 移植性
└── NFR-OPS: 運用性
```

---

## 2. NFR-PER: 性能要件

### NFR-PER-001: 起動時間

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-PER-001 |
| 要件名 | アプリケーション起動時間 |
| 説明 | アプリケーションが起動しメイン画面が表示されるまでの時間 |
| 基準 | データ件数 1000 件以下で 3 秒以内 |
| 優先度 | 推奨 |

### NFR-PER-002: データ保存応答時間

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-PER-002 |
| 要件名 | データ保存の応答時間 |
| 説明 | ユーザー操作後のデータ保存完了までの時間 |
| 基準 | データ件数 1000 件以下で 1 秒以内 |
| 優先度 | 推奨 |

### NFR-PER-003: 画面遷移応答時間

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-PER-003 |
| 要件名 | 画面遷移の応答時間 |
| 説明 | サイドバーボタンクリックからページ表示完了までの時間 |
| 基準 | 500ms 以内 |
| 優先度 | 推奨 |

## 3. NFR-REL: 信頼性要件

### NFR-REL-001: データ整合性

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-REL-001 |
| 要件名 | データの整合性保証 |
| 説明 | 正常操作においてデータの欠損・不整合が発生しないこと |
| 実現手段 | 操作ごとの即時保存、JSON シリアライズのバリデーション |
| 優先度 | 必須 |

### NFR-REL-002: エラー耐性

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-REL-002 |
| 要件名 | データファイル破損時のエラー耐性 |
| 説明 | データファイルが破損している場合でもアプリケーションがクラッシュせず起動すること |
| 実現手段 | load() での例外ハンドリング、空リストへのフォールバック |
| 優先度 | 必須 |

### NFR-REL-003: 操作確認

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-REL-003 |
| 要件名 | 破壊的操作の確認ダイアログ |
| 説明 | 削除操作など元に戻せない操作の前に確認ダイアログを表示すること |
| 対象 | アイテム削除（Inbox、見直し） |
| 優先度 | 必須 |

## 4. NFR-USA: 使用性要件

### NFR-USA-001: 日本語対応

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-USA-001 |
| 要件名 | UI の日本語対応 |
| 説明 | ラベル、ボタン、メッセージ等の UI テキストを日本語で表示すること |
| 対象フォント | Segoe UI, Yu Gothic UI, Meiryo UI, sans-serif |
| 優先度 | 必須 |

### NFR-USA-002: ダークテーマ

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-USA-002 |
| 要件名 | ダークテーマの適用 |
| 説明 | 目の疲労を軽減するダークテーマ（Catppuccin Mocha ベース）で表示すること |
| 優先度 | 必須 |

### NFR-USA-003: 視覚的フィードバック

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-USA-003 |
| 要件名 | 操作への視覚的フィードバック |
| 説明 | ホバー、アクティブ、無効状態などを視覚的に区別できること |
| 実現手段 | QSS によるボタン/入力フィールドの状態スタイリング |
| 優先度 | 必須 |

### NFR-USA-004: バッジによる状況把握

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-USA-004 |
| 要件名 | サイドバーバッジによる全体状況の把握 |
| 説明 | 各フェーズの未処理件数をサイドバーに表示し、ユーザーが状況を即座に把握できること |
| 優先度 | 必須 |

### NFR-USA-005: 必須入力の視覚表示

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-USA-005 |
| 要件名 | 必須入力項目の視覚的識別 |
| 説明 | 必須入力項目に `*` マークを表示し、バリデーションエラー時に赤色メッセージを表示すること |
| 対象 | 明確化フェーズのコンテキスト入力フォーム |
| 優先度 | 必須 |

### NFR-USA-006: ツールチップ

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-USA-006 |
| 要件名 | マトリクスビューのツールチップ |
| 説明 | マトリクス上のドット/ラベルにマウスホバーした際、詳細情報をツールチップで表示すること |
| 優先度 | 必須 |

## 5. NFR-SEC: セキュリティ要件

### NFR-SEC-001: ローカルデータ保存

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-SEC-001 |
| 要件名 | データのローカル保存 |
| 説明 | すべてのデータをローカルファイルシステムに保存し、外部サービスへの送信を行わないこと |

| 項目 | 内容 |
|------|------|

| 優先度 | 必須 |

## 6. NFR-MNT: 保守性要件

### NFR-MNT-001: 3 層アーキテクチャ

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-MNT-001 |
| 要件名 | Model-Logic-View の 3 層分離 |
| 説明 | データモデル、ビジネスロジック、GUI を分離し、各層が独立して変更・テスト可能であること |
| 優先度 | 必須 |

### NFR-MNT-002: テストカバレッジ

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-MNT-002 |
| 要件名 | ロジック層のテストカバレッジ |
| 説明 | ロジック層（models, repository, logic）のテストカバレッジが 80% 以上であること |
| 現状 | 99%以上 |
| 優先度 | 必須 |

### NFR-MNT-003: 型安全性

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-MNT-003 |
| 要件名 | 型ヒントと型チェック |
| 説明 | すべての関数に型アノテーションを付与し、mypy strict モードでエラーがないこと |
| 優先度 | 必須 |

### NFR-MNT-004: コード品質

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-MNT-004 |
| 要件名 | リンター・フォーマッターによる品質維持 |
| 説明 | ruff による lint/format チェックでエラーがないこと |
| 優先度 | 必須 |

### NFR-MNT-005: ログ出力

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-MNT-005 |
| 要件名 | 障害調査用のログ出力 |
| 説明 | すべてのモジュールで logging を使用し、ファイルにログを出力すること |
| 出力先 | logs/app_YYYY-MM-DD.log |
| ローテーション | 最大 10MB、30 世代保持 |
| 優先度 | 必須 |

### NFR-MNT-006: ドキュメント整備

| 項目 | 内容 |
|------|------|
| 要件 ID | NFR-MNT-006 |
| 要件名 | 設計書・仕様書・要件定義書の整備 |
| 説明 | 機能追加・変更時に関連ドキュメントを最新化すること |
| 優先度 | 必須 |
